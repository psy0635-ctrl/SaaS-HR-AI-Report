# LLM API 활용 방식 알아보기

> 파일명: `llm_api.md`

## 1. 개념

**LLM API**는 우리 서비스에서 대형 언어 모델을 직접 실행하지 않고, 외부 AI 모델 제공사의 서버에 요청을 보내 답변을 받아오는 방식이다.

쉽게 말하면, 백엔드가 AI 모델에게 다음처럼 요청하는 것이다.

```text
"다음 HR 데이터를 바탕으로 이번 달 근태 리포트를 작성해줘."
```

그러면 LLM API는 분석 결과를 자연어 문장으로 반환한다.

```text
"이번 달 개발팀의 평균 지각률은 8%이며, 전월 대비 2%p 감소했습니다..."
```

## 2. 왜 필요한가?

우리 팀이 직접 LLM을 학습하고 운영하려면 GPU, 모델 서빙, 보안, 비용 관리 등 복잡한 작업이 필요하다. 팀 프로젝트에서는 API를 사용하는 방식이 훨씬 현실적이다.

| 방식 | 설명 |
|---|---|
| 직접 모델 운영 | 서버에 LLM을 직접 설치하고 관리 |
| LLM API 사용 | 외부 API에 요청을 보내 결과를 받음 |

우리 프로젝트에서는 **LLM API를 사용해 AI Report 생성, HR 챗봇, 문서 요약** 기능을 구현할 수 있다.

## 3. 핵심 용어 정리

| 용어 | 설명 |
|---|---|
| API Key | API 사용자를 인증하는 비밀 키 |
| Prompt | LLM에게 전달하는 지시문 |
| Model | 사용할 AI 모델 |
| Token | LLM이 처리하는 텍스트 단위 |
| Context Window | 모델이 한 번에 참고할 수 있는 입력 길이 |
| Temperature | 답변의 창의성 또는 다양성을 조절하는 값 |
| Streaming | 답변을 한 번에 받지 않고 조금씩 받는 방식 |
| Structured Output | JSON처럼 정해진 형식으로 답변을 받는 방식 |
| Tool Calling | LLM이 외부 함수나 API 호출을 요청하는 방식 |

## 4. 기본 동작 흐름

```text
사용자
→ 프론트엔드에서 리포트 생성 요청
→ 백엔드 API
→ HR 데이터 조회
→ 프롬프트 생성
→ LLM API 호출
→ AI 리포트 응답 수신
→ DB 저장
→ 프론트엔드에 결과 표시
```

## 5. LLM API 요청 구조

일반적으로 LLM API 요청에는 다음 정보가 포함된다.

| 항목 | 설명 |
|---|---|
| model | 어떤 모델을 사용할지 지정 |
| input 또는 messages | 사용자 질문과 시스템 지시문 |
| temperature | 답변 스타일 조절 |
| max tokens | 최대 출력 길이 제한 |
| tools | 모델이 호출할 수 있는 외부 도구 |
| stream | 스트리밍 사용 여부 |

예시 구조:

```json
{
  "model": "example-llm-model",
  "input": [
    {
      "role": "system",
      "content": "너는 HR 데이터 분석 리포트를 작성하는 AI야."
    },
    {
      "role": "user",
      "content": "다음 데이터를 기반으로 개발팀 근태 리포트를 작성해줘."
    }
  ],
  "temperature": 0.3
}
```

## 6. 프롬프트 설계 방식

LLM API를 잘 쓰려면 프롬프트를 명확하게 작성해야 한다.

좋지 않은 예:

```text
리포트 써줘.
```

좋은 예:

```text
너는 HR 데이터 분석가다.
아래 JSON 데이터는 2026년 5월 개발팀의 근태와 성과 요약이다.

요구사항:
1. 핵심 요약을 3문장으로 작성한다.
2. 긍정적인 변화와 주의할 점을 나눈다.
3. 특정 직원을 단정적으로 평가하지 않는다.
4. HR 담당자가 바로 읽을 수 있는 보고서 말투로 작성한다.
```

프롬프트 구성 요소:

| 구성 요소 | 설명 |
|---|---|
| 역할 | AI가 어떤 역할인지 지정 |
| 입력 데이터 | 분석할 데이터 제공 |
| 출력 형식 | 표, 문단, JSON 등 지정 |
| 제약 조건 | 개인정보 보호, 단정 표현 금지 |
| 예시 | 원하는 답변 스타일 예시 제공 |

## 7. HR AI Report API 활용 예시

### 7.1 백엔드 처리 흐름

```text
POST /reports
→ 요청 파라미터 확인
→ HR 데이터 조회
→ pandas로 통계 계산
→ LLM 입력용 요약 JSON 생성
→ LLM API 호출
→ 결과 저장
→ report_id 반환
```

### 7.2 LLM 입력 예시

```json
{
  "company": "A회사",
  "department": "개발팀",
  "period": "2026-05",
  "kpi": {
    "late_rate": "8%",
    "absence_rate": "2%",
    "average_performance_score": 4.1,
    "average_stress_level": 4.2
  },
  "rules": [
    "개인 이름을 직접 언급하지 않는다.",
    "이직 가능성을 단정하지 않는다.",
    "개선 제안은 실행 가능한 형태로 작성한다."
  ]
}
```

### 7.3 출력 예시

```text
2026년 5월 개발팀은 전반적으로 안정적인 성과를 유지하고 있습니다.
평균 성과 점수는 4.1점으로 양호하지만, 평균 스트레스 수준이 4.2점으로 높게 나타났습니다.
특히 초과근무와 지각률을 함께 점검하여 업무 부담이 특정 구성원에게 집중되지 않도록 관리할 필요가 있습니다.
```

## 8. Structured Output 활용

리포트를 화면에 안정적으로 표시하려면 LLM에게 자유 문장만 받는 것보다 **정해진 JSON 형식**으로 받는 것이 좋다.

예시:

```json
{
  "title": "2026년 5월 개발팀 HR 리포트",
  "summary": [
    "성과 점수는 안정적인 수준입니다.",
    "스트레스 수준은 다소 높게 나타났습니다."
  ],
  "risks": [
    "초과근무 증가",
    "일부 구성원의 만족도 하락"
  ],
  "recommendations": [
    "업무 배분 점검",
    "1:1 면담 진행",
    "근태 추세 모니터링"
  ]
}
```

장점:

| 장점 | 설명 |
|---|---|
| 화면 표시 쉬움 | 제목, 요약, 추천사항을 각각 UI에 배치 가능 |
| 저장 쉬움 | DB 컬럼과 매핑 가능 |
| 오류 검증 가능 | 필수 필드가 있는지 확인 가능 |
| 재사용 가능 | PDF, 대시보드, 이메일에 활용 가능 |

## 9. Streaming 활용

AI 답변이 길면 사용자가 오래 기다리는 것처럼 느낄 수 있다. 이때 **Streaming**을 사용하면 생성되는 문장을 조금씩 화면에 보여줄 수 있다.

```text
일반 응답:
요청 → 오래 기다림 → 전체 답변 표시

스트리밍 응답:
요청 → 첫 문장 표시 → 다음 문장 표시 → 계속 표시 → 완료
```

우리 프로젝트에서는 `POST /chat`이나 긴 AI Report 미리보기에서 활용할 수 있다.

## 10. Tool Calling 활용

Tool Calling은 LLM이 직접 계산하거나 추측하지 않고, 필요한 도구를 호출하게 만드는 방식이다.

예시:

```text
사용자:
"개발팀의 이번 달 결근률 알려줘."

LLM:
get_attendance_summary(department="개발팀", period="2026-05") 호출 필요

백엔드:
실제 DB 조회 후 결과 반환

LLM:
"개발팀의 이번 달 결근률은 2%입니다."
```

Tool Calling을 사용하면 AI가 임의로 숫자를 만들어내는 문제를 줄일 수 있다.

## 11. 비용과 보안 관리

LLM API는 보통 사용량에 따라 비용이 발생한다. 따라서 다음을 관리해야 한다.

| 관리 항목 | 설명 |
|---|---|
| 입력 길이 | 불필요한 원본 데이터를 모두 보내지 않기 |
| 출력 길이 | 리포트 길이 제한 |
| 캐싱 | 같은 요청은 재사용 |
| 사용량 제한 | 사용자별 월간 리포트 생성 횟수 제한 |
| 로그 관리 | 요청과 응답 기록 저장 |
| 개인정보 보호 | 이름, 이메일 등 민감 정보 제거 |

## 12. 주의할 점

- **API Key를 코드에 직접 넣으면 안 된다.**
- LLM이 만든 숫자를 그대로 믿지 말고 계산은 백엔드에서 해야 한다.
- HR 개인정보는 가능한 한 익명화하거나 요약 통계로 전달해야 한다.
- 리포트 결과에는 생성 시점과 사용 데이터 범위를 표시하는 것이 좋다.
- 장애 상황을 대비해 API 실패 처리와 재시도 로직이 필요하다.

## 13. 우리 프로젝트 적용 정리

우리 프로젝트에서는 LLM API를 다음처럼 활용할 수 있다.

| 기능 | LLM API 활용 방식 |
|---|---|
| AI Report 생성 | HR 통계 요약을 입력으로 넣고 리포트 생성 |
| HR 챗봇 | 사용자 질문에 대해 RAG와 Tool Calling으로 답변 |
| 리포트 요약 | 긴 리포트를 3줄 요약으로 변환 |
| 문장 다듬기 | 관리자용 보고서 말투로 변환 |
| 추천사항 생성 | 근태/성과 지표를 바탕으로 관리 액션 제안 |

추천 구조:

```text
정확한 계산: 백엔드 + Python
문서 검색: RAG + 임베딩
자연어 생성: LLM API
실시간 답변: Streaming
정확한 데이터 조회: Tool Calling
```
