# 임베딩 알아보기

> 파일명: `embedding.md`

## 1. 개념

**임베딩(Embedding)**은 문장, 문서, 단어 같은 데이터를 AI가 이해하기 쉬운 **숫자 벡터**로 바꾸는 기술이다.

예를 들어 “퇴사 위험이 높은 직원”이라는 문장과 “이직 가능성이 큰 구성원”이라는 문장은 표현은 다르지만 의미가 비슷하다. 임베딩은 이런 의미적 유사성을 숫자 공간에서 가깝게 표현한다.

```text
"퇴사 위험이 높은 직원"
→ [0.12, -0.45, 0.88, 0.31, ...]

"이직 가능성이 큰 구성원"
→ [0.10, -0.42, 0.84, 0.35, ...]
```

두 벡터가 가까우면 의미가 비슷하다고 판단할 수 있다.

## 2. 왜 필요한가?

HR AI Report 기능에서는 사용자가 정확한 키워드를 입력하지 않아도 관련 정보를 찾아야 한다.

예를 들어 사용자가 “요즘 직원들이 힘들어하는 이유가 뭐야?”라고 물었을 때, 단순 키워드 검색은 “힘들다”라는 단어가 들어간 문서만 찾을 수 있다. 하지만 임베딩 검색은 “업무 부담”, “초과근무”, “만족도 하락”, “번아웃” 같은 의미적으로 관련 있는 문서도 찾을 수 있다.

| 필요한 이유 | 설명 |
|---|---|
| 의미 기반 검색 | 표현이 달라도 의미가 비슷하면 검색 가능 |
| RAG 구현 | 관련 문서를 찾아 LLM 답변에 넣을 수 있음 |
| 유사 질문 검색 | 이전 질문과 비슷한 질문을 찾을 수 있음 |
| 리포트 검색 | 기존 AI Report 중 관련 리포트를 찾을 수 있음 |
| HR 문서 검색 | 사내 규정, 평가 기준, 복지 문서 검색 가능 |

## 3. 핵심 용어 정리

| 용어 | 설명 |
|---|---|
| 임베딩 | 문장이나 문서를 숫자 벡터로 변환한 결과 |
| 벡터 | 여러 개의 숫자로 이루어진 배열 |
| 유사도 | 두 벡터가 얼마나 가까운지 나타내는 값 |
| 코사인 유사도 | 벡터 방향을 기준으로 비슷함을 계산하는 방식 |
| 벡터 DB | 임베딩 벡터를 저장하고 빠르게 검색하는 데이터베이스 |
| RAG | 검색한 문서를 LLM 답변에 활용하는 방식 |
| 청크 | 긴 문서를 검색하기 좋게 나눈 작은 단위 |
| 메타데이터 | 문서 제목, 작성일, 부서 등 부가 정보 |

## 4. 동작 흐름 또는 구조

RAG에서 임베딩이 사용되는 과정:

```text
HR 문서 수집
→ 문서를 작은 청크로 분리
→ 각 청크를 임베딩 벡터로 변환
→ 벡터 DB에 저장
→ 사용자 질문 입력
→ 질문도 임베딩 벡터로 변환
→ 비슷한 문서 검색
→ 검색 결과를 LLM에 전달
→ 답변 또는 리포트 생성
```

간단한 구조:

```text
사용자 질문
→ 임베딩 모델
→ 질문 벡터
→ 벡터 DB 검색
→ 관련 HR 문서 반환
→ LLM
→ 답변 생성
```

## 5. 사용 예시

HR 문서 검색 예시:

```text
사용자 질문:
"육아휴직 관련 규정 알려줘."

임베딩 검색 결과:
- 사내 복지 규정 문서
- 휴직 신청 절차 문서
- 근로기준법 관련 참고 문서

LLM 답변:
"육아휴직은 신청서 제출 후 HR 담당자 승인 절차를 거치며..."
```

AI Report 검색 예시:

```text
사용자 질문:
"지난 분기 이직 위험 관련 리포트 찾아줘."

검색 대상:
- 2026년 1분기 이직 위험 분석 리포트
- 개발팀 퇴사율 증가 원인 분석
- 만족도 하락 부서 리포트
```

키워드 검색과 임베딩 검색 차이:

| 구분 | 키워드 검색 | 임베딩 검색 |
|---|---|---|
| 검색 방식 | 같은 단어가 있는지 확인 | 의미가 비슷한지 확인 |
| 예시 | “퇴사”가 포함된 문서 검색 | “이직”, “조직 이탈”, “사직” 관련 문서도 검색 |
| 장점 | 빠르고 단순함 | 표현이 달라도 검색 가능 |
| 단점 | 단어가 다르면 못 찾을 수 있음 | 벡터 DB와 임베딩 모델 필요 |
| HR 적용 | 직원명, 부서명 검색 | 규정, 리포트, 질문 의미 검색 |

## 6. 장점

- 표현이 달라도 의미가 비슷한 정보를 찾을 수 있다.
- HR 문서, 리포트, FAQ 검색 품질을 높일 수 있다.
- RAG 기반 AI 챗봇을 만들 수 있다.
- 기존 리포트와 유사한 사례를 찾아 비교할 수 있다.
- 사용자가 정확한 용어를 몰라도 원하는 정보를 찾기 쉽다.

## 7. 주의할 점

- 임베딩 검색 결과가 항상 정답은 아니므로 LLM 답변에 출처를 표시하는 것이 좋다.
- 개인정보가 포함된 문서를 벡터 DB에 저장할 때는 암호화와 접근 제어가 필요하다.
- 문서를 너무 크게 나누면 검색 정확도가 떨어질 수 있다.
- 문서를 너무 작게 나누면 문맥이 부족해질 수 있다.
- 임베딩 모델이 바뀌면 기존 벡터를 다시 생성해야 할 수 있다.

## 8. 우리 프로젝트 적용 정리

우리 프로젝트에서는 임베딩을 HR 문서 검색, 리포트 검색, 유사 질문 검색에 활용할 수 있다.

| 적용 기능 | 활용 방식 |
|---|---|
| HR 문서 검색 | 사내 규정, 평가 기준, 복지 문서 검색 |
| AI Report 검색 | 과거 리포트 중 유사한 분석 결과 검색 |
| 유사 질문 검색 | HR 챗봇에서 이전 질문과 비슷한 답변 재활용 |
| RAG 챗봇 | 검색된 문서를 기반으로 신뢰도 높은 답변 생성 |
| 리포트 근거 제공 | AI가 참고한 문서나 데이터 출처 표시 |

추천 구조:

```text
HR 문서 / 리포트
→ 청크 분리
→ 임베딩 생성
→ 벡터 DB 저장
→ 사용자 질문
→ 유사 문서 검색
→ LLM 답변 생성
```
