# 소프트웨어 개발 문서 작성과 소통 방식

> 파일명: `software_documentation.md`

## 1. 개념

**소프트웨어 개발 문서**는 팀이 같은 목표를 이해하고 같은 기준으로 개발하기 위해 작성하는 자료이다.

개발 문서는 단순히 보고서를 예쁘게 쓰는 것이 아니다. 프로젝트에서 “무엇을 만들지”, “어떻게 만들지”, “누가 무엇을 맡을지”, “어떤 기준으로 완료를 판단할지”를 정리하는 도구이다.

팀 프로젝트에서 문서가 부족하면 다음 문제가 생긴다.

```text
기획자는 A라고 생각함
프론트엔드는 B라고 구현함
백엔드는 C라고 API를 만듦
AI 담당자는 필요한 데이터가 무엇인지 모름
결과적으로 기능이 서로 맞지 않음
```

## 2. 왜 필요한가?

우리 프로젝트는 SaaS, HR 데이터, AI Report, API, 프론트엔드, 백엔드가 모두 연결된다. 그래서 말로만 소통하면 놓치는 부분이 많다.

| 필요한 이유 | 설명 |
|---|---|
| 요구사항 정리 | 어떤 기능을 만들지 명확히 함 |
| 역할 분담 | 팀원별 작업 범위를 나눌 수 있음 |
| API 합의 | 프론트엔드와 백엔드 통신 규칙을 맞춤 |
| DB 합의 | 어떤 데이터가 필요한지 맞춤 |
| 일정 관리 | 마감일과 우선순위를 관리 |
| 유지보수 | 나중에 코드를 이해하기 쉬움 |

## 3. 핵심 문서 종류

| 문서 | 목적 | 작성 시점 |
|---|---|---|
| 프로젝트 개요서 | 프로젝트 목적과 배경 정리 | 시작 단계 |
| 요구사항 정의서 | 필요한 기능과 조건 정리 | 기획 단계 |
| 기능 명세서 | 기능별 동작 방식 정리 | 설계 단계 |
| 화면 설계서 | 화면 구성과 사용자 흐름 정리 | 설계 단계 |
| API 명세서 | 요청/응답 형식 정리 | 백엔드 설계 단계 |
| ERD / DB 설계서 | 데이터베이스 구조 정리 | DB 설계 단계 |
| AI 기능 설계서 | 프롬프트, 입력 데이터, 출력 형식 정리 | AI 기능 설계 단계 |
| 테스트 케이스 | 기능 검증 방법 정리 | 구현 후 |
| 회의록 | 결정사항과 할 일 기록 | 매 회의 후 |
| README | 실행 방법과 프로젝트 구조 안내 | 개발 중 계속 업데이트 |

## 4. 문서 작성 기본 원칙

| 원칙 | 설명 |
|---|---|
| 짧고 명확하게 | 긴 문장보다 표와 목록을 활용 |
| 결정사항 중심 | 논의 내용보다 최종 결정사항을 분명히 기록 |
| 예시 포함 | API, 화면, 데이터 예시를 넣으면 이해가 쉬움 |
| 최신 상태 유지 | 바뀐 내용은 문서도 함께 수정 |
| 담당자 표시 | 누가 맡았는지 기록 |
| 완료 기준 작성 | 언제 끝났다고 볼지 정의 |

나쁜 문서 예:

```text
리포트 기능을 잘 만든다.
```

좋은 문서 예:

```text
AI Report 생성 기능
- 사용자는 부서와 기간을 선택할 수 있다.
- 백엔드는 선택된 조건에 맞는 근태/성과 데이터를 조회한다.
- AI는 요약, 위험 요인, 추천 액션을 포함한 리포트를 생성한다.
- 생성된 리포트는 DB에 저장된다.
```

## 5. 요구사항 정의서 작성 방식

요구사항 정의서는 “무엇을 만들 것인가”를 정리하는 문서이다.

| 항목 | 예시 |
|---|---|
| 기능명 | AI Report 생성 |
| 사용자 | HR 담당자 |
| 목적 | 부서별 HR 데이터를 자동으로 분석하고 리포트 생성 |
| 입력 | 부서, 기간, 리포트 유형 |
| 처리 | HR 데이터 조회, 통계 계산, LLM 리포트 생성 |
| 출력 | 제목, 요약, 주요 지표, 추천 액션 |
| 완료 기준 | 리포트가 DB에 저장되고 화면에서 조회 가능 |

예시:

```md
## AI Report 생성 기능

### 목적
HR 담당자가 부서별 근태와 성과 데이터를 기반으로 AI 리포트를 생성할 수 있다.

### 입력값
- 부서
- 기간
- 리포트 유형

### 처리 흐름
1. 사용자가 리포트 생성을 요청한다.
2. 백엔드가 HR 데이터를 조회한다.
3. Python으로 KPI를 계산한다.
4. LLM API가 리포트 문장을 생성한다.
5. 결과를 DB에 저장한다.

### 완료 기준
- 생성된 리포트가 화면에 표시된다.
- 같은 리포트를 나중에 다시 조회할 수 있다.
```

## 6. API 명세서 작성 방식

API 명세서는 프론트엔드와 백엔드가 반드시 맞춰야 하는 약속이다.

| 항목 | 설명 |
|---|---|
| Method | GET, POST, PATCH, DELETE |
| URL | API 주소 |
| Request | 프론트엔드가 보내는 데이터 |
| Response | 백엔드가 반환하는 데이터 |
| Error | 실패 시 응답 |
| 권한 | 누가 호출할 수 있는지 |

예시:

````md
## POST /reports

### 설명
선택한 부서와 기간을 기준으로 AI Report를 생성한다.

### Request
```json
{
  "department": "개발팀",
  "period": "2026-05",
  "reportType": "attendance_performance"
}
```

### Response
```json
{
  "reportId": "R001",
  "title": "2026년 5월 개발팀 HR 리포트",
  "summary": "개발팀은 안정적인 성과를 유지하고 있으나..."
}
```
````

## 7. DB 설계 문서 작성 방식

DB 설계서는 어떤 데이터를 저장할지 정리하는 문서이다.

| 테이블 | 설명 |
|---|---|
| companies | 회사 정보 |
| users | 로그인 사용자 |
| employees | 직원 정보 |
| attendance_records | 근태 기록 |
| performance_reviews | 성과 평가 |
| ai_reports | AI 리포트 |

예시:

| 컬럼명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| employee_id | string | 직원 ID | Y |
| company_id | string | 회사 ID | Y |
| department | string | 부서 | Y |
| job_role | string | 직무 | N |
| hire_date | date | 입사일 | Y |

## 8. AI 기능 설계서 작성 방식

AI 기능은 일반 기능보다 입력과 출력이 더 중요하다. LLM은 같은 요청에도 다른 답변을 만들 수 있기 때문에, 형식을 명확히 정해야 한다.

| 항목 | 설명 |
|---|---|
| 기능명 | AI Report 생성 |
| 입력 데이터 | HR KPI JSON, 리포트 유형 |
| 프롬프트 | AI에게 전달하는 지시문 |
| 출력 형식 | Markdown 또는 JSON |
| 금지 표현 | 단정적 이직 판단, 개인정보 직접 노출 |
| 검토 방식 | HR 담당자 승인 후 저장 |

예시 출력 형식:

```json
{
  "title": "string",
  "summary": ["string"],
  "risks": ["string"],
  "recommendations": ["string"]
}
```

## 9. GitHub 이슈 작성 방식

이슈는 할 일을 추적하는 단위이다.

좋은 이슈 제목:

```text
[Backend] AI Report 생성 API 구현
```

이슈 본문 예시:

```md
## 작업 내용
- POST /reports API 구현
- 부서와 기간 기준 HR 데이터 조회
- LLM API 호출 결과 저장

## 완료 기준
- 정상 요청 시 reportId 반환
- 잘못된 부서 요청 시 400 에러 반환
- 생성된 리포트가 DB에 저장됨

## 참고 문서
- api_streamable_http.md
- llm_api.md
```

## 10. Pull Request 작성 방식

Pull Request는 코드 변경 내용을 팀원에게 설명하고 리뷰받는 공간이다.

PR에 포함하면 좋은 내용:

| 항목 | 설명 |
|---|---|
| 변경 내용 | 어떤 기능을 구현했는지 |
| 테스트 결과 | 어떤 테스트를 했는지 |
| 스크린샷 | 화면 변경이 있다면 첨부 |
| 관련 이슈 | 연결된 GitHub Issue |
| 리뷰 포인트 | 특히 봐줬으면 하는 부분 |

PR 템플릿 예시:

```md
## 변경 내용
- AI Report 생성 API를 추가했습니다.
- 리포트 생성 결과를 ai_reports 테이블에 저장합니다.

## 테스트
- POST /reports 정상 요청 확인
- 필수 파라미터 누락 시 에러 확인

## 리뷰 포인트
- API 응답 형식이 프론트엔드에서 사용하기 적절한지 확인 부탁드립니다.

## 관련 이슈
- closes #12
```

## 11. 회의록 작성 방식

회의록은 모든 대화를 적는 문서가 아니라, **결정사항과 할 일**을 기록하는 문서이다.

회의록 템플릿:

```md
# 회의록

## 회의 정보
- 날짜:
- 참석자:
- 주제:

## 논의 내용
- 

## 결정사항
-

| 담당자 | 작업 | 마감일 |
|---|---|---|
| 홍길동 | POST /reports API 구현 | 2026-05-22 |

## 다음 회의 전 확인할 것
- 
```

## 12. 팀 소통 방식

팀 프로젝트에서는 소통 규칙을 정해두면 충돌이 줄어든다.

| 상황 | 추천 소통 방식 |
|---|---|
| 기능 아이디어 | Notion 또는 회의록에 정리 |
| 개발 할 일 | GitHub Issue로 관리 |
| 코드 변경 | Pull Request로 공유 |
| API 변경 | API 명세서 먼저 수정 |
| DB 변경 | ERD 또는 DB 설계서 먼저 수정 |
| 긴급 오류 | 메신저로 알리고 Issue에 기록 |
| 결정 변경 | 회의록과 관련 문서 업데이트 |

## 13. 주의할 점

- 문서를 한 번 쓰고 방치하면 오히려 혼란이 생긴다.
- API나 DB 구조가 바뀌면 관련 문서를 반드시 같이 수정해야 한다.
- 구두로 결정한 내용도 회의록에 남겨야 한다.
- PR 설명 없이 코드만 올리면 리뷰가 어려워진다.
- AI 기능은 결과가 매번 달라질 수 있으므로 입력과 출력 기준을 문서화해야 한다.

## 14. 우리 프로젝트 적용 정리

우리 프로젝트에서는 다음 문서 세트를 만들면 좋다.

| 문서 | 추천 파일명 | 목적 |
|---|---|---|
| 프로젝트 개요서 | `project_overview.md` | 전체 목표와 범위 정리 |
| 요구사항 정의서 | `requirements.md` | 기능 목록과 완료 기준 정리 |
| API 명세서 | `api_spec.md` | 프론트엔드/백엔드 통신 규칙 |
| DB 설계서 | `db_schema.md` | HR 데이터 구조 정리 |
| AI 기능 설계서 | `ai_report_design.md` | LLM 입력/출력/프롬프트 정리 |
| 회의록 | `meeting_notes.md` | 결정사항과 할 일 기록 |
| README | `README.md` | 실행 방법과 문서 목록 안내 |

가장 중요한 원칙은 **기능을 만들기 전에 기준을 문서로 맞추고, 기능이 바뀌면 문서도 같이 바꾸는 것**이다.
