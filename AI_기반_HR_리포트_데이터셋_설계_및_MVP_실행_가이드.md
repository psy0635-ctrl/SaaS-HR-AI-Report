# AI 기반 HR 리포트 데이터셋 설계 및 MVP 실행 가이드

---

## 0. 문서 목적

본 프로젝트는 **SaaS HR 플랫폼 안에서 AI Report 기능을 개발하는 것**을 목표로 한다.  
관리자가 직원의 **근태, 성과, 이직률, 조직 데이터**를 직접 엑셀로 분석하지 않아도, 시스템이 자동으로 데이터를 분석하고 차트와 자연어 리포트 형태로 보여주는 구조를 만든다.

### 초보자용 읽기 가이드

이 문서는 데이터, 백엔드, 프론트엔드, AI 기능을 한 번에 다루기 때문에 처음 보면 용어가 많아 보일 수 있다.  
처음 읽을 때는 모든 기술 용어를 외우려고 하기보다, 아래 흐름만 먼저 이해하면 된다.

```text
1. 어떤 HR 데이터를 쓸지 정한다.
2. 그 데이터를 어떤 표 구조로 저장할지 정한다.
3. 실제 개인정보 대신 가짜 데이터를 만든다.
4. 만든 데이터를 화면과 AI 리포트에서 사용한다.
```

### 초보자용 핵심 용어 설명

| 용어 | 쉬운 설명 | 이 문서에서의 의미 |
|---|---|---|
| SaaS | 인터넷으로 사용하는 소프트웨어 서비스 | 회사가 설치하지 않고 웹에서 쓰는 HR 서비스 |
| HR | 인사 관리 | 직원, 부서, 근태, 성과, 퇴사 관련 업무 |
| AI Report | AI가 작성하는 분석 보고서 | 숫자 데이터를 사람이 읽기 쉬운 문장으로 요약한 리포트 |
| 데이터셋 | 분석에 사용할 데이터 묶음 | 직원 정보, 근태, 성과, 이직 데이터 모음 |
| 컬럼 | 표의 세로 항목 | `employee_id`, `age`, `department_id` 같은 항목명 |
| 테이블 | 같은 종류의 데이터를 모아둔 표 | 직원 표, 근태 표, 성과 표처럼 나눈 데이터 구조 |
| 스키마 | 테이블과 컬럼을 미리 정리한 설계도 | 어떤 테이블에 어떤 컬럼을 둘지 정한 약속 |
| 더미 데이터 | 실제가 아닌 연습용 가짜 데이터 | 개인정보 없이 만든 가상 직원 데이터 |
| row-level 데이터 | 한 줄이 한 사람 또는 한 기록을 뜻하는 데이터 | 직원 1명, 출근 기록 1건처럼 세부 단위의 데이터 |
| Kaggle | 예제 데이터셋을 찾을 수 있는 사이트 | HR 분석에 필요한 컬럼 구조를 참고하는 곳 |
| 공공통계 | 정부나 공공기관이 제공하는 통계 | 이직률, 근로시간, 임금 범위를 현실적으로 맞추는 참고 자료 |
| MVP | 최소 기능 제품 | 발표와 시연이 가능한 핵심 기능만 먼저 만든 버전 |
| 프론트엔드 | 사용자가 보는 화면 | 대시보드, 직원 목록, 리포트 화면 |
| 백엔드 | 화면 뒤에서 데이터를 처리하는 서버 | DB에서 데이터를 가져오고 API로 전달하는 부분 |
| API | 프론트엔드와 백엔드가 데이터를 주고받는 약속 | `/api/employees`처럼 데이터를 요청하는 주소 |
| CSV | 엑셀처럼 표 데이터를 저장할 수 있는 파일 형식 | 더미 데이터를 파일로 저장할 때 사용 |
| PostgreSQL | 데이터를 저장하는 데이터베이스 프로그램 | MVP에서 직원, 근태, 성과 데이터를 저장할 DB 후보 |
| LLM | 문장을 이해하고 생성하는 AI 모델 | HR 데이터를 자연어 리포트로 바꾸는 AI |
| 대시보드 | 핵심 지표를 한 화면에 모은 화면 | 직원 수, 이직률, 평균 성과를 차트로 보여주는 화면 |
| 이상 탐지 | 평소와 다른 수치를 찾아내는 것 | 갑자기 이직률이 오르거나 성과가 떨어지는 상황 감지 |
| 커밋 | 작업 내용을 Git에 저장하는 단위 | 문서 수정, 코드 추가 같은 변경 내역 기록 |
| 브랜치 | 작업을 나눠서 진행하는 Git 공간 | 데이터 작업, 백엔드 작업을 분리해서 진행하는 방법 |

## 1. 프로젝트 한 줄 요약

> **직원 정보, 근태, 성과, 이직 데이터를 기반으로 HR 지표를 자동 분석하고, 관리자에게 AI 자연어 리포트와 대시보드를 제공하는 SaaS HR 기능 개발 프로젝트이다.**

---

## 2. 왜 데이터셋 정리가 먼저 필요한가?

AI Report 기능을 만들려면 먼저 **어떤 데이터를 기준으로 분석할지**가 정해져야 한다.

예를 들어, 프론트엔드에서는 `이직률`, `평균 성과`, `초과근무 시간`을 차트로 보여줘야 하고, 백엔드에서는 해당 데이터를 API로 제공해야 한다.  
AI 분석 엔진은 이 데이터를 받아 자연어 리포트를 생성해야 한다.

따라서 팀원들이 각자 다른 컬럼명이나 다른 데이터 구조로 작업하면 나중에 기능을 합칠 때 문제가 생긴다.

### 데이터 기준이 없을 때 생기는 문제

| 문제 상황 | 예시 |
|---|---|
| 컬럼명이 서로 다름 | 한 명은 `employee_id`, 다른 한 명은 `emp_id` 사용 |
| 데이터 단위가 서로 다름 | 어떤 사람은 월별 데이터, 어떤 사람은 일별 데이터 사용 |
| 수치 범위가 비현실적임 | 결근율이 40%, 이직률이 80%처럼 생성됨 |
| 프론트와 백엔드 연동이 안 됨 | 프론트에서 필요한 값이 API에 없음 |
| AI 리포트가 부정확해짐 | 분석에 필요한 데이터가 부족하거나 의미가 불명확함 |

## 3. 최종 데이터 구성 전략

본 프로젝트에서는 실제 회사의 개인정보가 포함된 HR 데이터를 직접 수집하지 않는다.  
대신 아래 3단계 방식으로 데이터를 구성한다.

```text
1단계: Kaggle HR 데이터셋 참고
   ↓
2단계: 공공통계로 수치 범위 현실화
   ↓
3단계: Python / AI 스크립트로 row-level 더미 데이터 대량 생성
```

쉽게 말하면, 실제 직원 정보를 쓰지 않고 **참고 자료를 보고 그럴듯한 가짜 HR 데이터**를 만드는 방식이다.  
Kaggle에서는 "어떤 컬럼이 필요할지"를 보고, 공공통계에서는 "숫자가 어느 정도여야 현실적인지"를 확인한 뒤, Python으로 여러 명의 가상 직원 데이터를 만든다.

## 4. 데이터 생성 방향 요약

| 단계 | 내용 | 목적 |
|---|---|---|
| 1단계 | Kaggle HR 데이터셋 조사 | 어떤 컬럼이 필요한지 참고 |
| 2단계 | 공공데이터포털, 고용노동통계 조사 | 한국 기준 근로시간, 이직률, 임금 범위 참고 |
| 3단계 | 팀 공통 스키마 설계 | 프론트, 백엔드, AI가 같은 구조 사용 |
| 4단계 | Python으로 더미 데이터 생성 | 개인정보 없는 가상 HR 데이터 생성 |
| 5단계 | CSV 또는 PostgreSQL에 저장 | 실제 MVP 기능 개발에 사용 |
| 6단계 | AI Report, 대시보드, 이상 탐지에 활용 | 최종 서비스 기능 구현 |

## 7. Kaggle 데이터셋을 그대로 쓰면 안 되는 이유

Kaggle 데이터셋은 좋은 참고 자료이지만, 그대로 서비스 데이터로 쓰기에는 한계가 있다.

| 한계 | 설명 |
|---|---|
| 한국 기업 기준과 다를 수 있음 | 해외 데이터셋이므로 국내 노동시장 환경과 차이가 있음 |
| 근태 데이터가 부족할 수 있음 | 출근, 퇴근, 지각, 결근 같은 일별 근태 데이터가 없을 수 있음 |
| 월별 성과 흐름이 부족할 수 있음 | 시간 흐름에 따른 성과 추세 분석이 어려울 수 있음 |
| 서비스 DB 구조와 다를 수 있음 | 실제 SaaS 플랫폼에는 테이블을 나누어 저장해야 함 |
| 개인정보 이슈를 피해야 함 | 실제 직원 데이터가 아닌 가상 데이터 사용이 안전함 |

따라서 Kaggle 데이터는 **최종 데이터로 그대로 사용하기보다, 스키마와 분석 방향을 참고하는 용도**로 사용하는 것이 적절하다.

## 8. 공공통계 활용 방향

공공통계는 row-level 직원 데이터를 제공하지 않는 경우가 많다.  
그러나 더미 데이터를 만들 때 **수치 범위를 현실적으로 설정하는 기준**으로 활용할 수 있다.

여기서 헷갈리기 쉬운 점은 공공통계가 보통 "직원 한 명 한 명의 데이터"가 아니라 "전체 평균"을 제공한다는 것이다.  
예를 들어 공공통계에는 특정 직원 김OO의 근무시간이 아니라, "전체 근로자의 월평균 근로시간" 같은 값이 나온다. 이 문서에서는 그런 평균값을 참고해서 더미 데이터의 범위를 너무 과장되지 않게 맞춘다.

### 8.1 참고할 수 있는 공공 데이터 사이트

| 사이트 | 활용 목적 |
|---|---|
| 공공데이터포털 | 사업체노동력조사, 고용형태, 구인구직 통계 등 조사 |
| 고용노동통계 | 근로시간, 임금, 이직률, 입직률 등 노동 통계 확인 |
| KOSIS 국가통계포털 | 산업별, 지역별, 연령별 고용 통계 확인 |
| 고용24 | 채용, 구직, 취업 관련 흐름 참고 |

| 산업별 고용 규모 | 부서/직무 구성 참고 | 개발, 영업, 인사, 마케팅 등 구성 |
| 지역별 통계 | 지역 확장 시 참고 | 서울/경기 등 지역별 분포 설정 |

## 10. MVP에서 사용할 최종 데이터 테이블

처음부터 너무 복잡한 데이터베이스를 만들 필요는 없다.  
MVP에서는 아래 5개 테이블만 있어도 충분하다.

```text
departments
employees
attendance
performance
attrition
```

초보자 관점에서는 테이블을 **엑셀 시트**라고 생각하면 쉽다.  
예를 들어 `employees`는 직원 목록 시트, `attendance`는 출퇴근 기록 시트, `performance`는 성과 기록 시트이다. 각 시트는 `employee_id` 같은 공통 ID로 서로 연결된다.

---

## 11. 테이블 1: departments

부서 정보를 저장하는 테이블이다.

| 컬럼명 | 타입 | 설명 | 예시 |
|---|---|---|---|
| `department_id` | string | 부서 ID | D001 |

## 17. 더미 데이터 생성 규칙 초안

더미 데이터는 아무렇게나 만들면 안 된다.  
나중에 AI 리포트가 자연스럽게 나오도록 어느 정도 현실적인 패턴이 있어야 한다.

### 17.1 기본 생성 규모

| 항목 | 권장 값 |
|---|---|
| 직원 수 | 300~500명 |
| 부서 수 | 5개 |
| 기간 | 최근 6개월 |
| 근태 데이터 | 평일 기준 일별 생성 |
| 성과 데이터 | 월별 생성 |
| 이직 데이터 | 직원별 1개 생성 |

---

### 17.2 부서 구성 예시

| 부서 | 비율 예시 |
|---|---:|
| 개발팀 | 30% |
| 영업팀 | 25% |
| 고객지원팀 | 20% |
| 마케팅팀 | 15% |
| 인사팀 | 10% |

---

### 17.3 근태 생성 규칙 예시

| 항목 | 생성 기준 |
|---|---|
| 출근 시간 | 대부분 08:50~09:20 사이 |
| 퇴근 시간 | 대부분 18:00~20:00 사이 |
| 지각률 | 전체 근무일의 3~8% |
| 결근율 | 전체 근무일의 1~4% |
| 초과근무 | 평균 0~2시간 |
| 특정 부서 패턴 | 개발팀/영업팀은 초과근무가 조금 높게 설정 가능 |

---

### 17.4 성과 생성 규칙 예시

| 항목 | 생성 기준 |
|---|---|
| 성과 점수 | 0~100점 |
| 평균 점수 | 70~85점 |
| 우수 직원 | 90점 이상 |
| 위험 직원 | 60점 이하 |
| 부서별 차이 | 영업팀은 편차가 크고, 인사팀은 안정적으로 생성 가능 |

---

### 17.5 이직 위험 생성 규칙 예시

이직 위험 점수는 랜덤으로만 만들면 안 된다.  
아래 요인을 반영하면 AI Report에서 더 자연스러운 분석이 가능하다.

| 요인 | 위험 점수 영향 |
|---|---|
| 초과근무가 많음 | 위험 증가 |
| 성과 점수가 낮음 | 위험 증가 |
| 직무 만족도가 낮음 | 위험 증가 |
| 근속 기간이 짧음 | 위험 증가 |
| 결근/지각이 많음 | 위험 증가 |
| 워라밸 점수가 낮음 | 위험 증가 |

### 예시 로직

```text
risk_score = 기본 점수
           + 초과근무 영향
           + 성과 하락 영향
           + 결근/지각 영향
           + 만족도 영향
```

## 23. 백엔드 API 초안

프론트엔드와 AI 엔진이 사용할 API는 아래처럼 시작하면 좋다.

API는 "화면이 서버에게 데이터를 요청하는 주소"라고 이해하면 된다.  
예를 들어 직원 목록 화면이 `/api/employees`에 요청하면, 백엔드는 직원 목록 데이터를 보내준다. `GET`은 데이터를 조회할 때, `POST`는 새 리포트 생성처럼 서버에 작업을 요청할 때 사용한다.

| Method | Endpoint | 설명 |
|---|---|---|
| GET | `/api/dashboard/summary` | 대시보드 KPI 요약 |
| GET | `/api/departments` | 부서 목록 조회 |
| GET | `/api/employees` | 직원 목록 조회 |
| GET | `/api/employees/:id` | 직원 상세 조회 |
| GET | `/api/analytics/attrition` | 이직률 분석 조회 |
| GET | `/api/analytics/performance` | 성과 분석 조회 |
| GET | `/api/analytics/attendance` | 근태 분석 조회 |
| POST | `/api/reports/generate` | AI 리포트 생성 |
| GET | `/api/reports/:id` | 생성된 리포트 조회 |

---

## 24. AI Report 생성 흐름

```text
사용자가 기간/부서 선택
        ↓
백엔드가 DB에서 HR 데이터 조회
        ↓
AI 분석 엔진이 데이터 집계
        ↓
이상치 탐지 및 핵심 지표 계산
        ↓
LLM에게 요약 프롬프트 전달
        ↓
자연어 리포트 생성
        ↓
프론트엔드에 리포트와 차트 표시
```

이 흐름은 "사용자가 조건을 고르면, 서버가 데이터를 모으고, AI가 문장으로 요약해서, 화면에 보여준다"는 뜻이다.  
LLM은 숫자를 직접 DB에서 꺼내는 역할이 아니라, 백엔드가 정리해서 넘겨준 숫자를 읽고 자연스러운 보고서 문장으로 바꾸는 역할을 한다.

## 30. 데이터 파일 관리 규칙

### 30.1 raw 폴더

```text
data/raw/
```

원본 데이터를 저장한다.

`raw`는 "손대지 않은 원본"이라는 뜻이다.  
나중에 데이터가 이상해졌을 때 처음 받은 파일과 비교할 수 있어야 하므로, 이 폴더의 파일은 그대로 보관한다.

예시:

```text
kaggle_ibm_hr_attrition.csv
public_labor_statistics.csv
```

규칙:

- 원본 데이터는 직접 수정하지 않는다.
- 다운로드한 파일 그대로 보관한다.
- 수정이 필요하면 `processed` 폴더에 새 파일로 저장한다.

---

### 30.2 processed 폴더

```text
data/processed/
```

정제된 데이터를 저장한다.

`processed`는 "한 번 정리한 데이터"라는 뜻이다.  
예를 들어 컬럼명이 제각각인 파일을 프로젝트 기준 컬럼명으로 바꾸거나, 비어 있는 값을 정리한 파일을 여기에 둔다.

예시:

```text
ibm_hr_cleaned.csv
public_labor_statistics_cleaned.csv
```

규칙:

- 불필요한 컬럼 제거
- 컬럼명 통일
- 결측치 처리
- 분석에 필요한 형태로 정리

---

### 30.3 generated 폴더

```text
data/generated/
```

Python으로 생성한 최종 더미 데이터를 저장한다.

`generated`는 "프로그램으로 새로 만든 데이터"라는 뜻이다.  
Kaggle이나 공공통계 원본이 아니라, 우리 프로젝트 규칙에 맞춰 생성한 최종 가상 HR 데이터를 저장한다.

예시:

```text
departments_v1.csv
employees_v1.csv
attendance_v1.csv
performance_v1.csv
attrition_v1.csv
```

## 32. 커밋 메시지 규칙

커밋 메시지는 "이번에 무엇을 바꿨는지 적는 짧은 기록"이다.  
예를 들어 문서를 수정했다면 `docs`, 기능을 추가했다면 `feat`처럼 앞에 유형을 붙이면 팀원이 변경 내용을 빠르게 이해할 수 있다.

| 유형 | 의미 | 예시 |
|---|---|---|
| `docs` | 문서 작업 | `docs: 데이터셋 조사 문서 추가` |
| `data` | 데이터 파일 작업 | `data: HR 더미 데이터 v1 추가` |
| `feat` | 기능 추가 | `feat: AI 리포트 생성 API 추가` |
| `fix` | 오류 수정 | `fix: 이직률 계산 오류 수정` |
| `refactor` | 코드 구조 개선 | `refactor: 데이터 생성 로직 함수 분리` |
| `chore` | 설정 작업 | `chore: 프로젝트 폴더 구조 정리` |

## 36. 최종 결론

본 프로젝트의 데이터셋 방향은 다음과 같이 정리할 수 있다.

> **Kaggle HR 데이터셋의 스키마 구조를 참고하고, 공공통계 자료로 수치 범위를 현실화한 뒤, Python 또는 AI 스크립트를 활용하여 개인정보가 없는 row-level 가상 HR 데이터를 대량 생성한다.**

이 방식은 실제 개인정보 문제를 피하면서도, HR SaaS 플랫폼의 AI Report 기능을 충분히 구현하고 시연할 수 있는 현실적인 방법이다.

초기 MVP에서는 아래 기능에 집중한다.

1. HR 대시보드
2. AI 리포트 생성
3. 부서별 성과 분석
4. 이직 위험 직원 목록
5. 기본적인 PDF/Excel 내보내기

고급 기능인 HR 챗봇, 실시간 알림, 머신러닝 기반 예측 모델은 MVP 이후 확장 기능으로 두는 것이 좋다.

5. 이 데이터를 기반으로 대시보드, AI 리포트, 이직 위험 분석 기능을 구현한다.
