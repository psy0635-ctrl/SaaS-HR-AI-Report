## 1. SaaS 알아보기

**SaaS(Software as a Service)** 란 소프트웨어를 설치 없이 인터넷으로 구독해서 쓰는 방식. 사용자는 브라우저만 있으면 어디서든 접속 가능.

- HR SaaS 예시: flex, 그루비, 사람인HR — 근태·급여·인사관리를 웹에서 바로 씀. 기업이 서버를 따로 구축할 필요 없음.
- 보통은 기존 HR SaaS 플랫폼 위에 AI Report 모듈을 얹는 구조인데, 프로젝트 측에서 플랫폼을 제공해줄 가능성이 낮음.
- → **HR SaaS 플랫폼 자체도 우리가 직접 구현해야 함.** 완전한 멀티테넌시보다는 단일 회사 기준 MVP 먼저 만들고 추후 확장하는 방향.

**키워드:** 직접 구현 / React + Node.js / MVP 단일 테넌트 우선 / 멀티테넌시 추후

## 2. 데이터셋 확보 방향

공공데이터 포털은 집계 통계 위주라 **row-level 개인 단위 데이터가 없음.** 이직률·근로시간 등 현실적인 수치 범위 참고 정도로만 활용.

**Kaggle이 핵심.** 익명화된 합성 데이터지만 실제 HR 시스템과 유사한 컬럼 구조를 갖추고 있어 스키마 설계 참고에 최적.

### 주요 Kaggle 데이터셋

| 데이터셋 | 설명 |
|---|---|
| **IBM HR Analytics** | 이직 예측용. 부서·직급·성과점수·이직여부·근속년수·급여 등 1,470행. 가장 유명하고 컬럼 구조가 깔끔함. |
| **Employee/HR All-in-One** | 채용·교육·성과·근태·설문 통합 데이터. 여러 테이블 구조라 DB 설계 참고에 특히 좋음. |
| **Employee Performance & Productivity** | 생산성 점수·KPI 달성률·근무시간 위주. 성과 지표 중심. |
| **Human Resources Data Set** | 결근일수·성과평가·급여·부서·이직사유 포함. 컬럼 종류가 다양해서 대시보드 설계 참고용으로 좋음. |

**확정 방향:** Kaggle 스키마 구조 참고 → 공공통계로 수치 범위 현실화 → AI 스크립트로 row-level 더미 데이터 대량 생성.

**키워드:** Kaggle 위주 / 공공데이터 수치 보정 / 더미 데이터 생성 예정

## 3. LangChain / LangGraph 알아보기

- **LangChain** 은 LLM 호출·프롬프트·데이터 조회 등을 체인(chain)으로 묶어서 관리하는 프레임워크. 단순 API 호출보다 복잡한 파이프라인을 구조적으로 구성할 수 있음.
- **LangGraph** 는 LangChain 위에서 동작하는 에이전트 오케스트레이션 도구. 여러 LLM을 노드로 연결하고 상태를 관리하면서 흐름을 제어함.
- LangChain이 단일 파이프라인이라면, LangGraph는 **복수의 파이프라인이 조건에 따라 분기하고 합류하는 그래프 구조** 를 다룸.

**키워드:** LangChain — 단일 파이프라인 / LangGraph — 멀티 노드 오케스트레이션 / Python

## 4. 임베딩 알아보기

- **임베딩(Embedding)** 은 텍스트를 수백 차원의 숫자 벡터로 변환하는 기술. 의미가 비슷한 문장은 벡터 공간에서 가까워짐.
- 예시: "A팀 사람들 많이 쉬었어?" → 임베딩 변환 → "A팀 결근율"과 의미가 가깝다고 판단 → 맞는 데이터를 DB에서 찾아옴.
- 이걸 **RAG(Retrieval-Augmented Generation)** 라고 부름. 챗봇이 자연어 질의를 받아 어떤 데이터를 조회할지 판단하는 데 사용.
- MVP에서는 단순 키워드 매핑으로도 챗봇 구현 가능해서 **필수는 아님.** 고도화 단계에서 챗봇 정확도를 높이고 싶을 때 도입.

**키워드:** 벡터 변환 / RAG / 챗봇 고도화 단계 / MVP에선 선택

## 5. RESTful API / Streamable HTTP

### 비교

| 구분 | RESTful API | Streamable HTTP (SSE) |
|---|---|---|
| **방식** | 요청 1번 → 응답 1번. 처리 완료 후 전체 데이터를 한 번에 반환. | 응답을 조각조각 실시간으로 흘려보냄. 생성되는 즉시 프론트에 전달. |
| **특징** | 구현 단순. 응답 시간이 짧은 요청에 적합. | ChatGPT처럼 글자가 타이핑되듯 나오는 효과. LLM 응답에 최적. |
| **단점** | LLM처럼 생성에 수 초 걸리는 경우 응답 올 때까지 사용자가 그냥 기다려야 함. | 구현이 REST보다 복잡. 연결 유지 관리 필요. |

**확정 방향:** 둘 다 씀. 데이터 조회는 REST, AI 리포트·챗봇 응답은 SSE 스트리밍으로 혼용.

**키워드:** REST — 데이터 조회 / SSE — LLM 응답 / 혼용
