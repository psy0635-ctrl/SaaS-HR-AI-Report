# LangChain / LangGraph 알아보기

> 파일명: `langchain_langgraph.md`

## 1. 개념

먼저 **LLM(Large Language Model)**은 대량의 텍스트를 학습해 사람처럼 문장을 이해하고 생성하는 AI 모델이다. ChatGPT 같은 모델이 대표적인 예시다.

**LangChain**은 LLM을 실제 서비스에 쉽게 연결하기 위한 프레임워크이다. 단순히 질문하고 답하는 수준을 넘어, 데이터베이스 조회, 문서 검색, API 호출, 리포트 생성 같은 작업 흐름을 만들 수 있다.

**LangGraph**는 LangChain보다 더 복잡한 AI 작업 흐름을 **그래프 구조**로 제어하기 위한 도구다. 여러 단계가 있고, 조건에 따라 다음 작업이 달라지는 에이전트 구조에 적합하다.

| 구분 | 설명 |
|---|---|
| LLM | 문장을 이해하고 생성하는 대형 언어 모델 |
| LangChain | LLM 앱 개발을 쉽게 해주는 프레임워크 |
| LangGraph | 복잡한 AI 워크플로우를 그래프 형태로 구성하는 도구 |

## 2. 왜 필요한가?

우리 프로젝트의 AI Report 기능은 단순히 “리포트 써줘”라고 하는 기능이 아니다. HR 데이터를 조회하고, 필요한 통계를 계산하고, 결과를 요약하고, 사용자에게 자연어 리포트로 제공해야 한다.

이 과정에서 LangChain과 LangGraph가 필요하다.

| 필요 기능 | 설명 |
|---|---|
| 데이터 조회 | 직원, 근태, 성과 데이터를 DB에서 가져오기 |
| RAG | HR 문서나 기존 리포트를 검색해 답변에 활용 |
| Tool Calling | AI가 필요한 API나 DB 조회 도구를 호출 |
| Agent | AI가 스스로 필요한 작업 순서를 판단 |
| 리포트 생성 | 조회 결과를 기반으로 자연어 리포트 작성 |
| 챗봇 | HR 담당자가 질문하면 데이터를 조회해 답변 |

## 3. 핵심 용어 정리

| 용어 | 설명 |
|---|---|
| LLM | 대량의 텍스트를 기반으로 답변을 생성하는 AI 모델 |
| Chain | 여러 작업을 순서대로 연결한 흐름 |
| Agent | 목표를 보고 필요한 도구를 선택해 실행하는 AI 구조 |
| Tool Calling | AI가 외부 함수, API, DB 조회 도구를 호출하는 방식 |
| RAG | 외부 문서나 데이터를 검색해 AI 답변에 활용하는 방식 |
| Prompt | AI에게 전달하는 지시문 |
| State | 현재 작업 상태. 예: 사용자 질문, 조회 결과, 중간 분석 |
| Node | LangGraph에서 하나의 작업 단계 |
| Edge | LangGraph에서 다음 단계로 이동하는 연결 |

## 4. 동작 흐름 또는 구조

LangChain 기반 흐름:

```text
사용자 질문
→ Prompt 구성
→ HR 데이터 조회 Tool 실행
→ 조회 결과 정리
→ LLM 호출
→ AI Report 생성
→ 사용자에게 반환
```

LangGraph 기반 흐름:

```text
START
→ 질문 분석 Node
→ 필요한 데이터 판단 Node
→ DB 조회 Node
→ 통계 계산 Node
→ 리포트 생성 Node
→ 검토 Node
→ END
```

조건 분기 예시:

```text
사용자가 "이직 위험 분석" 요청
→ 이직 데이터 조회
→ 위험 요인 계산
→ 이직 위험 리포트 생성

사용자가 "근태 요약" 요청
→ 근태 데이터 조회
→ 부서별 지각/결근 통계 계산
→ 근태 리포트 생성
```

## 5. 사용 예시

HR AI Report 생성 예시:

```text
사용자:
"이번 달 개발팀의 근태와 성과를 요약해줘."

AI 처리 흐름:
1. 질문에서 "이번 달", "개발팀", "근태", "성과" 추출
2. 직원 DB에서 개발팀 직원 조회
3. 근태 DB에서 이번 달 출결 기록 조회
4. 성과 DB에서 평가 점수 조회
5. 주요 통계 계산
6. LLM이 자연어 리포트 생성
```

생성 결과 예시:

```text
개발팀의 이번 달 평균 출근 준수율은 94%이며,
전월 대비 지각 횟수는 12% 감소했습니다.
성과 점수는 평균 4.1점으로 안정적인 수준입니다.
다만 특정 직원군에서 초과근무가 증가하고 있어 업무 배분 점검이 필요합니다.
```

LangChain과 LangGraph 차이:

| 구분 | LangChain | LangGraph |
|---|---|---|
| 목적 | LLM 앱 구성 | 복잡한 에이전트 흐름 제어 |
| 구조 | 체인, 도구, 에이전트 중심 | 노드와 엣지 기반 그래프 |
| 난이도 | 비교적 쉬움 | 더 세밀한 설계 필요 |
| 적합한 기능 | 간단한 RAG, 챗봇, 리포트 생성 | 조건 분기, 반복 검토, 승인 흐름 |
| 우리 프로젝트 적용 | AI Report 기본 생성 | 복잡한 HR 분석 워크플로우 |

## 6. 장점

- LLM을 DB, API, 문서 검색 기능과 쉽게 연결할 수 있다.
- RAG를 적용해 더 정확한 답변을 만들 수 있다.
- Tool Calling으로 AI가 실제 데이터를 조회할 수 있다.
- LangGraph를 사용하면 복잡한 업무 흐름을 명확히 제어할 수 있다.
- HR 챗봇, 리포트 생성, 데이터 조회 자동화에 모두 활용 가능하다.

## 7. 주의할 점

- AI가 항상 정확한 답을 하는 것은 아니므로 **데이터 기반 검증**이 필요하다.
- 개인정보가 포함된 데이터를 LLM에 그대로 보내면 위험하다.
- 복잡한 Agent는 예측하기 어려운 동작을 할 수 있다.
- LangGraph를 처음부터 과하게 쓰면 개발 난이도가 올라간다.
- 리포트 생성 결과에는 출처 데이터와 생성 시간을 함께 남기는 것이 좋다.

## 8. 우리 프로젝트 적용 정리

우리 프로젝트에서는 초기에는 LangChain으로 기본 AI Report 기능을 만들고, 이후 복잡한 분석 흐름이 필요해지면 LangGraph를 적용하는 방식이 좋다.

| 적용 기능 | 추천 기술 | 설명 |
|---|---|---|
| AI Report 자동 생성 | LangChain | HR 데이터 요약 후 리포트 생성 |
| HR 챗봇 | LangChain + RAG | 사내 문서와 HR 데이터를 검색해 답변 |
| 데이터 조회 자동화 | Tool Calling | AI가 필요한 DB/API 조회 |
| 복잡한 분석 흐름 | LangGraph | 이직 위험 분석, 검토, 재생성 흐름 제어 |
| 관리자 승인 흐름 | LangGraph | AI 리포트 생성 후 사람이 검토 |
